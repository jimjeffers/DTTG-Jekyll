I've studied user experience and interaction design from the perspective of software applications and web sites, but never truly considered the parallels and great illustrations of user experience design successes and failures that occur in video games. I guess it should seem obvious since a game itself is a very robust and rich software project. Let's take a look at games and their design in depth.

What is a Game?

A game is an activity with rules. It is a form of play often but not always involving conflict, either with other players, with the game system itself, or with randomness/fate/luck.

Before we can dissect the design of a game we need to understand what a game itself is. The IGDA has provided a set of rules in their curriculum framework and concede that:

  • Most games have goals, but not all (e.g. The Sims, SimCity).
  • Most games have defined start and end points, but not all (World of Warcraft, Dungeons & Dragons).
  • Most games involve decision-making on the part of the players, but not all (e.g. Candyland, Chutes & Ladders).

Game design is the process of crafting a system of play in which players' actions have meaning in the context of the game environment. Game design encompasses the set of principles, concepts, and practices that lead to the development of a high-quality product.

Implicit in the process of game design is the consideration of design trade-offs to allow the implementation of a game in some human playable interactive environment.

The Atomic Parts of a Game

According to the IGDA, game design can be broken down into the following atomic parts:

  • Game objects (tokens) and game setting
  • Rules
  • Dynamics
  • Play mechanics – What are the rules? How do players interact with the environment? Do they provide discrete input (e.g. puzzle games) or is react to a randomness/chance (e.g. slot machines)
  • Goal(s) –
  • Conflict
  • Theme/Color

Let’s focus on Game Mechanics

Good mechanics of a game are key. This is where we determine how the player interacts with the game environment. It’s also our seek balance. If we make the game too simple or too difficult for players it’s likely we’ll ruin the experience. I suspect, that the case more often than not is we come across a game that has been made too difficult by a series of design flaws. Parallels in Game Design and Software Applications

There are several areas in game design that relate specifically to software design.

Providing Adequate Feedback to Players

In a game a player may require constant feedback as to what the input they have provide is producing. In a software application this also occurs but at a more passive level. A good example of this would be crisis points in your software. This is how you handle errors in user input. Defensive Design for the Web, a 'classic' book by 37 Signals, highlights how we provide feedback (or the lack there of) at these critical points of interaction with users.

Teaching Your Players how to Play the Game

Games tend to be much more varied than software in a sense that the designers have a lot more freedom to create a unique experience. As a result games come with directions and tutorials. Users come into the game expecting to learn it the first time through. With software we're still grappling with that. For a number of years 37 Signals has made a big fuss about 'blank slate' pages that are used in their applications. Yahoo! has taken this a step further and for a number of years now has provided interactive tours guiding you through the process of using some of their pages.
